# Desktop Readiness

ClaraVision starts as a web console, but the first version should remain ready
to be wrapped by a desktop app later.

## Boundary

The interface owns rendering and interaction only.

It should not directly own Memoria data, Continuity data, microphone handling,
desktop permissions, or agent execution.

## Browser Mode

When no desktop bridge exists, the page reads `data/state.json` first and only
falls back to local demo data when the snapshot is missing or invalid.

This keeps the first version easy to run in a browser and easy to verify.

## Electron Mode

The first Electron shell exposes this bridge before loading the page:

```js
window.ClaraVisionBridge = {
  async getState() {
    return state;
  },
  async refresh() {
    return updatedState;
  }
};
```

For the first desktop prototype, the bridge reads the same static
`data/state.json` snapshot as browser mode. Later, the desktop shell can collect
data from Memoria, Continuity, and agent runtimes outside the page, then pass the
combined display state into ClaraVision.

This lets the same interface run in:

- a normal browser
- an Electron desktop window
- a later floating desktop assistant

## Transparent Orb Mode

`npm run desktop` opens the desktop shell as a transparent always-on-top
orb overlay. The window uses the full display bounds instead of a small fixed
rectangle or the app work area, while the rendered brain core remains circular
inside the canvas.

Current controls:

- hover a node to see a short summary
- click a node to focus it
- double-click or press Esc to reset
- press O to switch view mode
- press R to refresh the static snapshot
- press S to cycle idle / reading / thinking
- press B to cycle brightness
- press C to cycle transparent contrast presets
- press + to cycle orb size
- press P to pause or resume animation
- press Cmd/Ctrl + Shift + Space to show or hide the desktop overlay
- press Cmd/Ctrl + Shift + I to cycle mouse interaction modes
- right-click for refresh, reset, mode switch, state, brightness, contrast, size, pause, pinning, mouse interaction, opacity, and quit

## Data Source Plan

The current first version uses `data/state.json` when available, generated by
`scripts/build_state.py`. Local demo data in `app.js` is only the fallback.

The snapshot uses Memoria recall for relevance, Memoria public export with
archived records for visual density, and Memoria stats for the corpus count.
Private memories are converted into restricted shadow particles for density;
their content is not placed into the desktop visual layer.

Later versions should map real sources into the same state object:

- Memoria: shared observable facts
- Continuity: current continuation position
- Clara: Clara-facing conversation and relationship context that is allowed to be shown
- Lara: Lara task state and tool activity that is allowed to be shown
- Codex: engineering task state, local tool activity, and write-back events

ClaraVision should render those sources. It should not become their storage
owner.

## Later Desktop Features

Keep these outside the first desktop prototype:

- microphone permission
- speech recognition
- speech output
- background agent execution
- native notifications

Those features belong in the desktop shell, not in the visual rendering layer.
